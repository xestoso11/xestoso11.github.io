Atos


Motivación para en entrar a la empresa

La finalidad por la que entre en esta empresa fue para tener una base en desarrollo y una visión diferente a la que tiene un técnico de sistemas, que me permitiera poder trabajar en un puesto con perfil devops con más facilidad.


Inicios y evolución

En esta empresa realize un bootcamp para convertirme en desarrollador full stack, en la que aprendí a usar lenguajes como java, typescript y permitió mi mejora en html y css. En toda la parte de backend usamos como lenguaje java, además de spring boot para facilitar el desarrollo con sus clases, interfaces y anotaciones ya precreadas, como base de datos usamos h2 una base de datos en memoria. En el frontend usamos el framework angular, además de node, también diferentes librerias de css como bootstrap y SweetAlert2, que nos facilito mucho el trabajo para el diseño de la interfaz de usuario.


Terminado el bootcamp, los tutores conformaron equipos de seis personas con las que tuvimos que aprender a trabajar conjuntamente, para desarrollar una aplicación desde cero, a través de unos requisitos indicados por ellos. Inicialmente generamos unas tareas principales para el desarrollo de esta aplicación, que a su vez fuimos dividiendo en subtareas, en las que estipulamos unos tiempos para cada tarea. En el desarrollo de esta aplicación tuvimos que aprender a buscar apis externas que realizaran algunas de las funciones que nos pidieron, e a implementarlas en nuestro código de forma que funcionaran correctamente.


¿Porque no me permanecí en la empresa?

Durante las prácticas de ciclo que fueron aproximadamente 3 meses, me indicaron desde un primer momento que no se iba a tocar nada relacionado con sistemas, y que estaría todo completamente relacionado con desarrollo de aplicaciones web, algo que yo no estaba buscando.

Emetel


Motivación para en entrar a la empresa

La razón por la que me uni a esta empresa fue para empezar a trabajar en un puesto relacionado con sistemas y que me permitiera continuar aprendiendo, aparte de que se trataba de unha sección que se estaba comenzando a formar, por lo que habría muchas oportunidades de futuro.


Inicios y evolución

Inicie mi puesto como implantador de una aplicación para diferentes puerto de españa, la aplicación se llama Ágata.

Gracias a está aplicación aprendí a trabajar con activos, propiedades de activos, plantillas, alertas, usuarios, grupos de usarios, asignación de permisos de estos, y protocolos para enviar las alertas a través de diferentes medios de comunicación.


La aplicación que más uso en el día a día es mobaxterm, y la creación de tuneles a través de ella para tener conectividad o probarla con dispositivos y aplicaciones, todas las máquinas virtuales son de alguna distro de linux, tambien uso mRemoteNG, cuando se tratan de máquinas con interfaz gráfica, todas windows, y anydesk para alguna máquina muy concreta.

A parte del uso de diferentes aplicaciones de vpn, forticlient, checkpoint y Pulse Secure para conectarme a las diferentes máquinas virtuales que usamos para implantar la aplicacion.


Aprendí a motorizaban diferentes dispositivos a través de protocolos como modbus, snmp, icmp, peticiones rest, tanto desde la propia aplicación como usando aplicaciones externas, desde aplicaciones externas para determinar que no fuese problema de conectividad o que el dispositivo no estuviese funcionando correctamente.

  • Para motorizar activos a través de modbus, como aplicaciones externas usabamos Modbus Poll o radzio, para comprobar la conectividad, o si los valores de las diferentes direciones eran los correctos.
  • Para la motorizar de activos a través de snmp, para leer los mibs y buscar los oid que nos podrían interesar para motorizar usabamos ireasoning, después de encontrar la propiedad que nos interesara, sacábamos el oid y le hacíamos un snmpwalk o snmpget, dependiendo si el oid nos fuese a devolver 1 único valor o varios valores, para comprobar que esa direción existe y nos da un valor coherente, para después motorizarla desde la aplicación de la empresa.
  • Para el protocolo icmp, haciendo un ping se comprobar si teniamos conectividad.
  • Para las peticiones rest, podiamos probar desde un curl, o mirar si la direción existia con un navegador web, hasta hacer un tunel con mobaxterm y comprobarlo desde postman

Usamos bases de datos como MySQL, Monogo y Postgress, para hacer cambios o mirar datos usamos aplicaciones como DBeaver,mongodb compass, mongochef, o accediendo directamente por terminal.


Aparte de la aplicación Ágata, se monto un prometheus para poder tener métricas de la aplicación y de las máquinas en donde estaba alojada, para que nos avisase si algo fallaba, para recolectar las métricas de las máquinas usamos node_exporte, para las métricas de los contenedores docker cadvisor, mongo_exporter y mysql_exporter para las métricas de las bases de datos, Blackbox exporter para recolectar métricas de endpoinds, para apache apache_exporter, para kafka y zookeper JMX Exporter, y para recolectar las métricas de los micros de la aplicacions usamos el propio prometheus, y para poder graficar todos los datos grafana, y para las alertas alermanager junto a msteams, para poder tener tanto alertas de email como por equipos de teams. Todo esto estaba montado en un docker compose.

En docker también montamos un redpanda console para revisar los topics y consumers de los micros dentro de kafka.


Llegamos a desarrollar una api entera en nodejs, para implementar una funcionalidad dentro de la aplicación.


Además de crear tareas programadas tanto en linux como en windows, y creación de script en bash, Powershell, Python, AutoIt.
